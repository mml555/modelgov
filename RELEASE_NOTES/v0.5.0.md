# Ai-Guard v0.5.0

**Tag date:** 2026-07-01

First aligned, pinnable release. All packages move to a single `0.5.0` line
(API, CLI, policy-engine, TS SDK, Python SDK, create-ai-guard). Builds on the
v0.0.0 baseline with a wave of operability, trust, and correctness work — the
flat, file-config path remains the default and every new capability is
opt-in / behind a flag.

## What's new since v0.0.0

### Trust & correctness
- **Safety cost reserved upfront** — the input-safety classifier cost is now
  included in the budget reservation, not just settled after, so model+safety
  can't overshoot a cap.
- **`config_hash` + `policy_version` on every request log** — operators can
  correlate any decision to the exact config that produced it (surfaced in
  `GET /v1/requests/:id`).
- **Richer SDK errors** — `AiGuardError` now exposes `reasonCode`,
  `auditRequestId`, `budgetRemaining`, `feature`, `userType`,
  `resolvedModelClass`.

### Operability
- **DB-backed API keys** — issue / rotate / revoke without redeploy
  (`/v1/admin/keys`, `ai-guard keys`), hashes only at rest.
- **Tamper-evident admin audit log** — hash-chained; `/v1/admin/audit` +
  `/verify`; wired to key, policy, and erasure mutations.
- **Versioned policy store** (opt-in) — validate → activate → rollback, audited.
- **GDPR/CCPA erasure endpoint** + per-feature retention.
- **Response streaming (SSE)**, **OpenTelemetry export**, and a **secrets-manager
  (`*_FILE`) convention** — all opt-in.

### Enterprise (opt-in, default-off — see docs/design/)
- Operator **SSO/OIDC + RBAC**, and **hierarchical budgets** (node tree, atomic
  multi-level reservation, counter sharding, tenant-bound keys + per-tenant
  policy). Behind flags; the flat path is unchanged.

### Docs
- New: [routing & block-vs-degrade](../docs/routing.md), HA architecture,
  disaster recovery, threat model + SOC 2 mapping, versioning policy, and design
  notes for the larger tracks.

## Routing behavior (unchanged, now documented)

Class order is fixed **cheap → standard → premium**; primary on allow/degrade,
fallback on provider failure; budget-aware degrade downgrades one permitted
class. Full decision table in [docs/routing.md](../docs/routing.md).

## Deploy notes

- Run migrations on deploy (through `0015_request_log_policy_version.sql`).
- All new subsystems are **off by default**: `API_KEYS_DB_ENABLED` aside, set
  `POLICY_STORE_ENABLED`, `HIERARCHICAL_BUDGETS`, `OIDC_*`,
  `OBSERVABILITY_PROVIDER=otel` only when you want them.
- Nothing changes the flat file-config path a v0.0.0 deploy relied on.
